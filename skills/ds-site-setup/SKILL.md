# Stardust Design System Site — Setup & Maintenance Skill

This skill covers two tasks:

1. **Phase A — Site scaffolding**: Generating or regenerating the 5 site shell files
   (`index.html`, `main.css`, `tokens.css`, `nav.js`, `.nojekyll`)
2. **Phase B — Nav and index maintenance**: Adding a new component to the nav and the
   homepage component grid

---

## Site architecture overview

```
stardust4/
  .nojekyll                     ← Disables Jekyll on GitHub Pages
  docs/
    index.html                  ← Landing page + component grid
    assets/
      css/
        main.css                ← Site chrome: --xp- vars, layout, shared UI classes
        tokens.css              ← Stardust --sd- token CSS vars (Figma-synced)
      js/
        nav.js                  ← Global nav; hardcoded strings ONLY — no user input to innerHTML
    components/
      [component].html          ← One file per component (generated by ds-component-doc skill)
    tokens/
      colour.html
      typography.html
      spacing.html
      radius.html
```

**GitHub Pages config:** Source = `main` branch, Folder = `/docs`
**Site root:** `https://masonmaddy.github.io/stardust4/`
**All content pages** sit exactly one folder deep, so every page references:
`../assets/css/main.css` and `../assets/js/nav.js`

## Phase A — Site scaffolding

### When to use

Use Phase A when:
- Setting up a new fork or clone of the repo (first time)
- Recovering from a corrupted/lost shell file
- Regenerating `tokens.css` after Figma token changes (most common use)

### `tokens.css` regeneration (most common task)

The token sync workflow:

1. Use `get_variable_defs` (Figma MCP) on node `1:5354` in file `a7JnfZ0Nd8df1TBPaMQ5Tj`
2. Use `use_figma` to enumerate all variables from:
   - `base` collection (VariableCollectionId:3794:373) — primitives
   - `mapped` collection (VariableCollectionId:3794:458) — semantic aliases
3. Resolve all alias values to their primitive hex values
4. Regenerate `docs/assets/css/tokens.css` following the structure:
   - Section 1: TIER 1 — Primitives (raw hex values, `--sd-colour-[palette]-[step]` etc.)
   - Section 2: TIER 2 — Semantic (alias vars, `var(--sd-colour-[primitive])`)
5. Commit the updated file — all HTML pages pick up changes automatically

**CSS variable naming convention:**
- Replace `/` with `-` in Figma token paths
- Add `--sd-` prefix
- Examples: `colour/action/primary` → `--sd-colour-action-primary`
- Examples: `spacing/component/md` → `--sd-spacing-component-md`
- Examples: `radius/m` → `--sd-radius-m`

**Important:** Never change CSS variable names once they're used in component pages.
The variable name is the stable contract. Only the value changes.

**Key brand note:** Primary action colour is teal `#00776B` (colour/action/primary).
Coral `#FF5A35` is Xplor corporate brand for `--xp-coral` (site chrome only, in main.css).

### `main.css` regeneration

Only needed if the site chrome (nav, layout, shared classes) needs to be rebuilt.
`main.css` uses `--xp-*` vars (Xplor corporate brand: coral `#FF5A35`, navy `#1A2B4A`).
It is NOT Figma-synced — it's hand-authored site chrome that changes rarely.

The `--xp-` custom properties must always be present at `:root`:
```css
--xp-coral: #FF5A35;
--xp-coral-hover: #E04D2C;
--xp-navy: #1A2B4A;
--xp-navy-light: #243556;
--xp-off-white: #F7F8FA;
--xp-border: #ECEEF2;
--font-heading: 'Sora', sans-serif;
--font-body: 'Inter', sans-serif;
```

### `nav.js` regeneration

`nav.js` generates the global nav via JS injection. All nav content is hardcoded
strings — no user input or URL data ever touches `innerHTML`. This must be preserved.

The file structure:
```js
// SECURITY comment at top explaining the constraint
var BASE_PATH = '/stardust4';
var COMPONENT_LINKS = [ /* hardcoded list */ ];
var TOKEN_LINKS = [ /* hardcoded list */ ];
// buildNav(), inject(), initScrollSpy(), initCopyButtons() functions
```

## Phase B — Nav and index maintenance

### When to use

Every time a new component page is generated by the `ds-component-doc` skill, two
additional updates are required:

1. **Add to `nav.js` `COMPONENT_LINKS` array** — one new object
2. **Add to `docs/index.html` component grid** — one new card

### Step B1 — Update nav.js

Open `docs/assets/js/nav.js` and add one entry to `COMPONENT_LINKS`:

```js
var COMPONENT_LINKS = [
  { label: 'Button', href: BASE_PATH + '/docs/components/button.html' },
  // Add new component below:
  { label: 'Badge',  href: BASE_PATH + '/docs/components/badge.html' },
];
```

Rules:
- `label`: Display name exactly as it appears in Figma (e.g. "Badge", "Input", "Modal")
- `href`: Must use `BASE_PATH + '/docs/components/[filename].html'` — no hardcoded paths
- Alphabetical order within the list is preferred but not required
- `BASE_PATH` = `/stardust4` — update this if the repo is renamed

### Step B2 — Update index.html component grid

Open `docs/index.html` and add a new card inside `.cards-grid`:

```html
<a href="components/badge.html" class="component-card">
  <div class="component-card__preview">
    <style>
      /* Preview uses --sd- token vars — never hardcoded hex */
      .idx-badge-preview { /* ... */ }
    </style>
    <span class="idx-badge-preview" aria-hidden="true">Badge</span>
  </div>
  <div class="component-card__info">
    <div class="component-card__name">Badge</div>
    <div class="component-card__desc">Status labels, counts, and indicators.</div>
  </div>
</a>
```

Also update the stat bar count:
- Find `<div class="stat__value">1</div>` (the component count stat)
- Increment by 1 for each new component added

### Step B3 — Commit

```
git add docs/assets/js/nav.js docs/index.html
git commit -m "feat: add [Component] to nav and index"
git push
```

## Security constraint

`nav.js` uses `innerHTML` exactly zero times. All DOM construction uses
`document.createElement()`, `setAttribute()`, and `createTextNode()`.
This is intentional and must be preserved. The security comment at the top of
`nav.js` documents this for future contributors. **Do not change this pattern.**
