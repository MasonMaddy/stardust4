# Source intake checklist

What to capture per source type so the research report has clean provenance. The goal is enough
to *cite*, not a full transcript.

## Stakeholder idea
- Who raised it and their role.
- The problem **as they framed it** (their words, not your reframing).
- Why now / what triggered it.
- Any assumed solution — note it, but keep problem and solution separate.

## Canny feature requests
- Pull via the Canny MCP read tools (read-only) — see `canny-intake.md`.
- Per post: title, the underlying problem, vote/score, status, board, notable comments.
- Look for **clusters** — several posts describing one underlying need.

## Customer interviews / conversations
- Date, segment/persona (not name unless kept deliberately), and context of the conversation.
- **Verbatim quotes** for anything you'll cite — paraphrase loses signal.
- Separate what the customer *did/experienced* from what they *asked for*.

## Support / sales / internal signals
- Volume and recency where available.
- Link to the source system rather than copying bulk content.

## Existing Jira / Confluence context
- Search before writing — is there prior discovery, a related epic, or a duplicate?
- Use the Atlassian MCP **search/fetch** tools (read-only). Link, don't duplicate.

---

**Quality bar:** if a source is thin or missing, say so explicitly in the report. A gap named is
useful; a gap hidden is a risk. Never manufacture evidence to fill a section.
