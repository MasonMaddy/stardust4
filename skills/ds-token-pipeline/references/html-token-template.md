# Token Reference Page — HTML Template

How to assemble a token reference page at `docs/tokens/<category>.html`. The live
exemplars are `docs/tokens/colour.html` (swatch pattern) and `docs/tokens/spacing.html`
(scale-bar + table pattern) — copy their structure for every new category page.

## Page skeleton

Token pages use the shared site assets — never standalone self-contained HTML.

```html
<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="UTF-8">
  <meta name="viewport" content="width=device-width, initial-scale=1.0">
  <title>[Category] Tokens — Stardust Design System</title>
  <link rel="stylesheet" href="../assets/css/main.css">
  <link rel="stylesheet" href="../assets/css/tokens.css">
  <style>
    /* Page-chrome/demo styles ONLY — see rules below */
  </style>
</head>
<body>
  <div id="site-nav"></div>
  <div class="site-content">
    <div class="page-layout">
      <main class="main-content">
        <!-- page-header, then sections -->
      </main>
      <aside class="page-toc" aria-label="On this page">
        <p class="page-toc__title">On this page</p>
        <ul class="page-toc__nav"><!-- one <li><a href="#id"> per section --></ul>
      </aside>
    </div><!-- /.page-layout -->
  </div><!-- /.site-content -->
  <script src="../assets/js/nav.js"></script>
</body>
</html>
```

## Page header

First child of `<main>`, doubles as the "Overview" TOC target:

```html
<div class="page-header scroll-spy-target" id="overview">
  <p class="page-header__eyebrow">Token Foundation</p>
  <h1 class="page-header__title">[Category]</h1>
  <div class="page-header__meta">
    <span>Synced from Figma YYYY-MM-DD</span> <span>·</span> <span>[collections / base unit]</span>
  </div>
  <p><!-- 2–3 sentence intro: the two-tier model as it applies to this category --></p>
</div>
```

## Sections

Each section is `<section class="section scroll-spy-target" id="...">` with an `<h2>`.
Standard order (every `id` gets a matching TOC entry):

1. **Semantic tokens** (`id="semantic"`) — the tokens components actually use, first.
2. **Primitives** (`id="primitives"` / `id="base"`) — raw values, with a note that they
   are referenced by semantic tokens only, never used directly in components.
3. **Category-specific extras** — e.g. colour has a Site chrome (`--xp-`) section with a
   warning callout that those are NOT Stardust tokens.
4. **DTCG JSON** (`id="dtcg"`) — last. A `callout callout--info` pointing at the canonical
   export `tokens/stardust.tokens.json` (generated by `scripts/build-tokens-json.mjs`),
   then an illustrative excerpt in a `.code-block` with `<span class="code-block__lang">`.

## Token display patterns

Pick per category — these are the only listing patterns in use:

- **Swatch rows** (colour): a `.palette-group` per intent group with a
  `.palette-group__header` (+ `.palette-group__count`); each token is a `.swatch-row` =
  `.swatch-block` (inline `background:` — the hex here is *data being documented*, not
  styling) + `.swatch-info` (`.swatch-name`, `.swatch-token` showing
  `figma/token/name → alias` with `.alias-arrow`) + trailing `.swatch-hex`.
- **Scale bars** (spacing, radius): a `.scale-group` with a `.col-headers` grid header
  and `.spacing-row`s — step, value, visual bar (`.scale-bar` sized to the value), CSS var.
- **`ds-table` + `token-pill`** (semantic tokens with usage guidance): columns
  Token / Alias / Value / CSS Variable / Use.
- **Live previews** (radius, motion): a small demo element styled *via the token itself*
  (`border-radius: var(--sd-radius-m)`), so the page proves the value renders.

Every table/row shows **both names**: the Figma token (`spacing/component/md`) and the
CSS variable (`--sd-spacing-component-md`) — mapping: `/` → `-`, prefixed `--sd-`.

## Rules (the page must obey the repo's hard rules)

- [ ] Inline `<style>` is **page chrome only** (row layout, hovers, group borders) and
      references `--sd-*` tokens for its own styling — never hardcode chrome colours.
- [ ] Never redefine a `ds-*` component rule inline (architecture guard fails the build).
      If the page uses a `ds-*` component, `<link>` its CSS from `assets/css/components/`.
- [ ] Token values come from `tokens.css` / the Figma `base` + `mapped` collections —
      Stardust primary is teal `#00776B`; never coral in token tables or swatches.
- [ ] Raw hexes are allowed only as *documented data* (swatch backgrounds, `.swatch-hex`
      labels, DTCG snippets) — `lint-hex.mjs` guards component CSS, not doc pages, but
      don't leak data hexes into chrome styles.
- [ ] Give every `<button>` a `type`; keep HTML valid (`html-validate` runs in CI).

## Assembly checklist

1. Copy the skeleton; set `<title>`, eyebrow, `<h1>`, meta (sync date, collections).
2. Build sections in the standard order with the right display pattern per tier.
3. Cross-check every value and CSS var name against `docs/assets/css/tokens.css`.
4. Add the TOC entries — one per `scroll-spy-target` id, in page order.
5. Link the page in `nav.js` `TOKEN_LINKS` (that's a `ds-site-setup` task, not ad-hoc).
6. If tokens changed, run `node scripts/build-tokens-json.mjs`; then run the local CI
   suite (`check-links.mjs`, `check-token-refs.mjs`, `check-architecture.mjs`, html-validate).
