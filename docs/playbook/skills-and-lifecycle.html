Skills & the Lifecycle

The Product Development Lifecycle was drawn long before AI could help run it. This page maps the skills in this repo onto that process — where they already automate the manual work, and where the gaps and opportunities still are.

✦
The operating principle. AI automates the manual parts — gathering, collating, drafting, decomposing — to free people for what is irreducibly human: creativity, judgement, and direct customer contact. Every skill keeps a human in the loop and checks in at each step. AI clears the path to the creative and customer-facing work; it never replaces it.

Skills mapped to the lifecycle

Reading down the stages of the lifecycle, here is what each skill covers today and where its output lands. Skills in the upstream product pipeline mostly write to Atlassian (Confluence / Jira); the design-system skills write to the repo.

Delivery (plan / implement) — gap — dev-handoff feeds engineers and build agents an agent-readable spec, but no skill yet runs the build/PR loop.
Deployment · QA & Rollout — gap — Repo CI checks guard the design system, but no skill covers test generation, QA, or rollout.

Documenting any of the above ds-page-author, ds-site-setup Narrative pages, nav, and the index grid (repo) — cross-cutting.

The component pipeline runs in four phases — review → spec → sandbox → build — and is held to the highest review bar. The upstream product pipeline (product-research → product-brief) sits before design and mostly writes to Atlassian. Every pipeline now ends in a default-on review gate (sandbox-review, proto-design-review, handoff-review, brief-review) that follows the same pattern: deterministic pre-checks, two adversarial reviewers, a Blocker / Should-fix / Nit report, and a binary verdict before the work moves on.

Gaps & remediation roadmap

The strongest coverage today is in the middle of the lifecycle — research, brief, component design, prototype, and handover. The clearest opportunities to automate more manual work are at the front edge (prioritisation) and the back half (delivery, QA, rollout, and measurement), plus closing the feedback loop.

Analysis & Research

Assisted prioritisation

The Impact × Confidence × Ease score is computed by hand today. A skill could draft the scoring from the research evidence and Canny signal for a human to adjust — keeping the judgement call with the PM.

Delivery

Build & PR loop

A delivery skill could consume the dev-handoff agent-readable spec and open scoped, reviewable PRs against acceptance criteria — the engineer stays the author and reviewer.

QA & Rollout

Test generation & checks

A QA skill could generate test cases from the brief's success metrics and run automated accessibility / visual-regression checks before the Ready-to-Go gate.

Rollout

Release comms drafting

Release notes, support articles, and training-video scripts are all manual at the Released-Software gate. Drafting them from the brief and changelog (the Canny changelog integration is a start) would free the team for the launch itself.

Measure Success

Success-measure review

A skill could assemble the one-month review — pulling analytics against the metrics defined in Discovery — so the team spends its time interpreting, not collating.

Feedback loop

Close the loop to research

What's learned post-release should flow back into product-research via Canny — turning released software into the next round of validated signal.

⚠️
This is a roadmap, not a commitment. These are candidate directions for where AI could clear manual work in the lifecycle. Each would be built as a skill on the product/ track, behind the same human-in-the-loop gate as every existing skill.
