Sync Pipeline

How design system changes flow from Figma to code — what's automated today, and the planned end-to-end automation with human review gates.

Today — the token sync chain

Tokens already flow from Figma to a published, machine-readable export:

Figma (Stardust Components)
  → tokens.css           (re-synced; names stable, values updated)
  → build-tokens-json.mjs
  → stardust.tokens.json (committed; CI fails if it drifts)
  → consumers re-fetch on release

The same pattern backs the Component API export. See Token Architecture for the token model and consumption details.

Planned — Figma → DS automation

🚧
Not yet built. The flow below is the agreed design for automating component/token updates end to end. It is documented here so the intent is captured; the automation is a later phase.
  1. Designer updates the Figma library and publishes it.
  2. Figma fires a LIBRARY_PUBLISH webhook to a lightweight endpoint (e.g. a GitHub Actions receiver).
  3. The endpoint triggers a Claude Code run.
  4. Claude Code pulls the diff via the Figma MCP — what changed.
  5. Component-review skills run: token changes, new/modified components, breaking vs non-breaking.
  6. Claude Code generates DS-repo changes (tokens, docs, snippets) and opens a PR.
  7. Engineers review and merge.
  8. On merge → tag a release → Renovate/Dependabot opens version-bump PRs in consumer repos.

The last mile — does a DS merge update apps?

Whether a merged DS change reaches an engineering codebase automatically depends on how the DS is distributed:

Distribution methodAuto-update?
NPM / Swift Package / Maven with a semver rangeYes — on next install with ^version
Git submoduleManual pull required
Copy-paste / local filesManual — no auto path
Monorepo (shared packages/)Live immediately after merge

Recommended: engineers receive a Renovate/Dependabot PR that bumps the DS version — they choose when to land it, keeping control over when breaking changes arrive.
