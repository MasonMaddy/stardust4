Larger devices — big phone & tablet adaptation

Big phones keep the edge-to-edge sheet. On tablets the sheet is capped at 640px and centred (M3's max sheet width; iPad natively replaces sheets with form sheets/popovers) — beyond that width, prefer a dialog or the desktop dropdown pattern instead of a sheet.

Pattern guidance — when to use what

SituationUseWhy
Native iOS / Android appPlatform sheet container (UISheetPresentationController / M3 ModalBottomSheet) with DS-styled content insideFree detents, drag physics, accessibility and keyboard avoidance; users get platform-correct behaviour
Date / time selection on mobileNative OS pickersPer the Input spec — OS pickers beat any custom sheet for these
Mobile web / PWA / webviewStardust sheet (this proposal)No native container exists; DS sheet keeps tokens, Cards and behaviour consistent
TabletSheet centred at max-width 640px — or prefer a dialogM3 caps sheet width at 640dp; iPad natively swaps sheets for form sheets/popovers
Desktop webNo sheet — dropdown menu or dialogSheets are a thumb-reach pattern; the Phase 2 Select dropdown is the desktop equivalent
Single-select, instant effect, low riskAuto-close on pick, no header buttonsFewest taps; scrim/Escape cover abandonment
Single-select with consequencesSelection + Confirm CTA + X closeReview before commit; X = leave without applying
Multi-selectCheckbox cards + Apply (n) CTA + Clear + X closeBatch commit; live count sets expectations
X close buttonRequired when a CTA exists; optional on instant sheetsVisible, focusable dismiss for keyboard/SR users; distinct from Confirm
