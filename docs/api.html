Component API

Every Stardust component is published as a machine-readable manifest — so AI agents and non-web tooling can read a component's classes, variants, tokens, and props without parsing the demo-heavy doc HTML. The files are derived from the component CSS and doc pages, committed to the repo, and kept in sync by CI.

The files

All three are served from the site root and regenerate together:

FileWhat it is
/llms.txt Flat llmstxt.org-style entry point — point an agent here first. Links every component + the index.
/api/index.json Catalogue of all components: name, title, version, root class, and a link to each manifest.
/api/<name>.json Per-component manifest (full detail). Example: api/button.json.

Manifest schema

Each api/<name>.json contains:

FieldDescription
name / titleFile stem and display name.
versionLatest changelog version from the doc page.
rootClassThe component's base class, e.g. .ds-btn.
stylesheet / docUrlPath to the shared CSS and the human doc page.
dependenciesOther components this one composes (links their CSS).
classes / elements / modifiersEvery .ds-* selector, BEM __elements, and --modifiers.
states / stateHooksCSS pseudo-states used and .is-* hooks.
tokensUsedEvery --sd-* token the component consumes — a component → token dependency graph.
sectionsThe doc page's section outline (id + title) — your index into the prose.
changelogFull version history.
propsThe Figma ↔ code mapping table, where the page has one.

Consuming it

List components, then load one manifest:

// 1. Catalogue of all components
const index = await (await fetch(
  'https://masonmaddy.github.io/stardust4/api/index.json'
)).json();

// 2. Load one component's full manifest
const entry  = index.components.find(c => c.name === 'button');
const button = await (await fetch(
  'https://masonmaddy.github.io/stardust4/' + entry.api
)).json();

button.rootClass;    // ".ds-btn"
button.modifiers;    // ["destructive", "full", "ghost", "icon-only", ...]
button.tokensUsed;   // ["--sd-colour-action-primary", "--sd-radius-lg", ...]
button.props;        // Figma property → code prop mapping
🤖
For AI agents Point the agent at /llms.txt — it links every component manifest and the token export, and is ~95% smaller than scraping the doc pages.

The contract

  • Derived, not hand-written. Generated from the component CSS + doc pages by scripts/build-component-api.mjs. Never edit api/*.json or llms.txt by hand.
  • CI-guarded. A --check step fails the build if the export drifts from source, so what's published always matches the components.
  • Stable names. Token and class names are the contract; values may change on a Figma re-sync, names don't.
  • Coverage. props tables exist where a component documents one; message-box and bottom-sheet are not yet fully covered.

Maintaining the pipeline is documented in the ds-component-api skill.
