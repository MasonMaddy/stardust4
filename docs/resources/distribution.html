Distribution Architecture

Published

How Stardust components move from design spec to npm packages that product teams install and ship.

Two repos, one system

Stardust ships through two repositories with distinct responsibilities:

Stardust4

Design authority

This site. Tokens, component CSS, Figma sync, API specifications, review gates, and documentation. A no-build static site deployed via GitHub Pages. Changes here define what a component looks like, how it behaves, and which tokens it uses.

stardust-components

Implementation layer

Stencil web components that consume the design specs and produce framework-specific npm packages. Monorepo with pnpm workspaces, Lerna independent versioning, and CI/CD for build, test, and publish.

i
Why separate? Stardust4 is a no-build static site with CI guards designed around that identity. A Stencil build toolchain with workspaces, versioning, and npm publishing is a fundamentally different concern. Separate repos keep the boundary clean.

Packages

Four npm packages ship under the @xplor-education scope:

Package Contents Audience
stardust-core Stencil web components with scoped CSS and tokens Any web project
stardust-vue Auto-generated Vue 3 proxies with v-model Vue 3 projects
stardust-vue-compat Vuetify adapter — :rules, color, legacy props Office teams migrating from Vuetify
stardust-react Auto-generated React wrappers with event bindings React projects

Update pipeline

Design changes flow from Stardust to consumers through a sync-and-validate pipeline:

1. Design change merges

A Stardust4 PR merges — updated component CSS, tokens, or API JSON deploys to GitHub Pages.

2. Sync

bash 
node scripts/sync-stardust.mjs

Pulls updated tokens, CSS, and API JSONs into the implementation repo. Reports what changed.

3. Validate & build

bash 
node scripts/validate-contracts.mjs   # CI fails if contract not met
pnpm build                             # builds all 4 packages
pnpm test                              # runs spec tests

validate-contracts.mjs checks every modifier and token in the API JSON is implemented in the Stencil component. If Stardust adds a modifier, CI fails until the component implements it.

4. Publish

A tagged release triggers the CI release workflow, which publishes new package versions to npm via Lerna.

Key decisions

Scoped CSS, not Shadow DOM

Stencil components use scoped: true. Stardust CSS is BEM-based for the light DOM (.ds-btn, .ds-btn--solid) — it works directly in scoped mode. --sd-* tokens on :root inherit naturally. DevTools show familiar ds-* classes. Tailwind and other utility frameworks compose freely alongside.

Vendored sync, not submodules

A sync script pulls files from Stardust4 via GitHub Pages (or a local path). Files are committed to the repo. No submodule complexity, no network dependency at build time, explicit version control. CI validates freshness.

Contract validation via API JSON

Each component publishes a machine-readable API JSON (docs/api/*.json) defining modifiers, elements, states, and tokens. The validate script asserts that the Stencil implementation covers everything the spec declares. Design decisions flow into CI enforcement automatically.

Vuetify compat as a separate package

stardust-vue-compat wraps the auto-generated Vue proxies with Vuetify-shaped prop surfaces. Teams that do not need Vuetify compatibility skip this dependency entirely.

Versioning

Packages use independent semver via Lerna:

Packages Strategy
core + vue + react Lockstep — a core change bumps all three
vue-compat Independent — the adapter surface evolves separately
i
Design versions vs package versions Component version numbers on this site (in each changelog table) are design versions. npm package versions are implementation versions. The sync script reports which design version is vendored so the mapping is always traceable.
