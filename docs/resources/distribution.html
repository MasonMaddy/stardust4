Distribution Architecture

Published

Why two repos

Stardust ships through two repositories with distinct responsibilities:

  • Stardust4 (this site) is the design authority — tokens, component CSS, Figma sync, API specifications, and review gates. It is a no-build static site deployed via GitHub Pages.
  • stardust-components is the implementation layer — Stencil web components that consume the design specs and produce framework-specific npm packages for product teams.

Stardust defines what a component looks like, how it behaves, and which tokens it uses. stardust-components implements that definition as real, packaged code that teams install and ship.

Packages

Four npm packages ship under the @xplor-education scope:

Package What it contains Who should use it
@xplor-education/stardust-core Stencil web components (<sd-button>, <sd-input>, etc.) with scoped CSS and Stardust tokens baked in. Any web project — works with or without a framework.
@xplor-education/stardust-vue Auto-generated Vue 3 proxies with v-model support. Vue 3 projects that want typed, reactive components.
@xplor-education/stardust-vue-compat Vuetify-shaped adapter layer. Accepts :rules, color, and legacy props (dense, outlined, etc.) and maps them to Stardust equivalents. Office teams migrating from Vuetify — drop-in replacement that preserves existing prop patterns.
@xplor-education/stardust-react Auto-generated React wrappers with event bindings. React projects.

How the pipeline works

Design changes flow from Stardust to consumers through a sync & validate pipeline:

  1. A Stardust4 PR merges — new component CSS, tokens, or API JSON deploys to GitHub Pages.
  2. In stardust-components: run node scripts/sync-stardust.mjs to pull updated files.
  3. The sync script reports what changed (e.g. "button: new modifier outline").
  4. Developer updates the Stencil component to implement the new contract.
  5. CI validates contracts (validate-contracts.mjs), builds all packages, runs tests.
  6. A tagged release publishes the new versions to npm.

The contract validation is the enforcement mechanism — if Stardust adds a modifier or token to a component's API JSON, CI fails in stardust-components until the Stencil component implements it. This ensures the packaged code always matches the design spec.

Key architectural decisions

Scoped CSS, not Shadow DOM

Stencil components use scoped: true, not shadow: true. Stardust CSS is BEM-based for the light DOM (.ds-btn, .ds-btn--solid) — it works directly in scoped mode. --sd-* tokens on :root inherit naturally. DevTools show familiar ds-* classes. Tailwind and other utility frameworks compose freely alongside the components.

Vendored sync, not submodules

A sync script pulls tokens.css, component CSS, and API JSONs from Stardust4 (via GitHub Pages or a local path). Files are committed to the implementation repo. No submodule complexity, no network dependency at build time, explicit version control. CI validates they are not stale.

Contract validation via API JSON

Each Stardust component publishes a machine-readable API JSON (at docs/api/*.json) defining modifiers, elements, states, and tokens used. The validate-contracts.mjs script asserts that the Stencil implementation covers every modifier and references every token. This is how design decisions flow into CI enforcement automatically.

Vuetify compat as a separate package

stardust-vue-compat wraps the auto-generated Vue proxies with Vuetify-shaped prop surfaces — :rules validation, color mapping, accepted-but-ignored legacy props. Teams that do not need Vuetify compatibility skip this dependency entirely.

Versioning

Packages use independent semver via Lerna:

  • stardust-core, stardust-vue, and stardust-react release in lockstep — a core change bumps all three.
  • stardust-vue-compat versions independently — the adapter surface evolves separately from the underlying components.

Stardust4 component version numbers (shown in the changelog on each component page) are design versions. npm package versions are implementation versions. The sync script reports which design version is vendored so the mapping is always clear.

Storybook

The stardust-components repo includes a Storybook instance with interactive stories for every component. This serves as the playground for testing the real packaged components — while the pages on this site remain the design spec and documentation.

Storybook builds deploy to GitHub Pages from the stardust-components repo. Each component page in this site links to its Storybook story for the interactive version.
