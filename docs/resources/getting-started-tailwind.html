Getting Started — Tailwind

Published

How Stardust components and tokens work alongside Tailwind CSS in your project.

1. Why it works out of the box

Stardust components render in the light DOM using scoped CSS (not Shadow DOM). This means:

  • Tailwind utility classes compose freely on the outside of a Stardust component
  • Component internals are protected by BEM scoping — Tailwind resets and utilities don't leak in
  • --sd-* token custom properties on :root are accessible from Tailwind's arbitrary value syntax
i
No special setup required. If your project already uses Tailwind and you install a Stardust package (stardust-vue, stardust-react, or stardust-core), everything works immediately.

2. Use Stardust tokens in Tailwind

Reference --sd-* tokens directly via Tailwind's arbitrary value syntax:

html 
<div class="p-[var(--sd-spacing-4)] rounded-[var(--sd-radius-md)]
            bg-[var(--sd-colour-surface-default)]
            border border-[var(--sd-colour-border-default)]">
  <p class="text-[var(--sd-colour-text-primary)]
             font-[var(--sd-font-family)]">
    On-system card using only Tailwind utilities + Stardust tokens
  </p>
</div>

This keeps your layout and spacing on-system without writing any custom CSS.

3. Layout with Tailwind, components from Stardust

The recommended pattern: use Tailwind for layout, spacing, and page-level styling, and Stardust components for interactive UI elements.

vue 
<template>
  <div class="flex flex-col gap-[var(--sd-spacing-4)] max-w-md mx-auto">
    <SdInput v-model="name" label="Child name" />
    <SdInput v-model="email" label="Email" type="email" />

    <div class="flex gap-[var(--sd-spacing-3)] justify-end">
      <SdButton type="ghost">Cancel</SdButton>
      <SdButton type="solid">Save</SdButton>
    </div>
  </div>
</template>

4. Optional: Tailwind preset

A Tailwind preset that maps --sd-* tokens to Tailwind's theme (so you can write bg-sd-surface instead of bg-[var(--sd-colour-surface-default)]) is planned as a future addition to the stardust-core package.

i
Not required to get started. The preset is a convenience layer. Arbitrary value syntax works today with zero configuration. The preset will be published as @xplor-education/stardust-core/tailwind-preset when ready.

5. What to avoid

Don't Why Do instead
Override ds-* classes with Tailwind Breaks component internals and won't survive version updates Use component props to control appearance
Hardcode hex values for colours Won't track token changes or future theme support Use var(--sd-colour-*) via arbitrary values
Rebuild Stardust components with Tailwind utilities Duplicates the system and drifts from spec Import the component from the npm package
