Component

Date Picker

WIP v1.0.0 Updated 2026-07-07

One calendar surface for choosing a single date, presented three ways: an always-visible inline card, a text input with an anchored popup (type a date or pick one), and — on phones — the same grid inside the shipped bottom sheet. It replaces the legacy InputDatePicker + CustomDatePicker + InlineDatePicker trio, which duplicated calendar logic three ways and disagreed on min/max semantics.

The inline calendar surface — the same grid is reused unchanged inside the popup and the phone sheet.

ℹ️
Min / max are inclusive. min and max stay selectable — only days strictly before min or strictly after max disable. disablePast is the same rule pinned to today, and keeps today selectable. This is stated once, on purpose: the three legacy pickers documented min/max inconsistently (one treated max as exclusive), and that ambiguity is exactly why they collapse into this one component.

Mode: phone (bottom sheet)

On phones the anchored popup would clip against the viewport, so mode="sheet" presents the same calendar grid inside the shipped Bottom Sheet. The trigger is a read-only .ds-input__box--select showing the chosen date; tapping it opens a .ds-sheet (role="dialog", aria-modal="true") whose .ds-sheet__list holds the grid in --fluid mode (columns stretch to fill; day targets never drop below 44px) and whose .ds-sheet__footer carries a Done button. The scrim dismisses; focus returns to the trigger on close. No date-picker CSS is added for this mode — it is purely the calendar surface composed into bottom-sheet.css. (Live demo omitted here because the sheet is a fixed full-viewport overlay; see the workbench and the Bottom Sheet page.)

4Usage Guidelines

✓ Do

  • Use mode="inline" when the calendar is the primary task (a booking screen, a date filter)
  • Use mode="input" in forms where a date is one field among many, so typists can key it fast
  • Switch to mode="sheet" on phones — the popup clips against small viewports
  • State a real min/max when a range is genuinely bounded, and remember both ends are selectable
  • Pass events to surface days that already carry bookings or activity

✗ Don't

  • Reach for the legacy InputDatePicker/CustomDatePicker/InlineDatePicker — they're replaced by this one component
  • Use this for choosing a time — that's the Time Picker; or a date range — not in v1.0
  • Disable a day without a reason a user can infer — pair it with helper text explaining the bound
  • Re-implement the calendar grid to add month/agenda views — that's the deferred ds-calendar
  • Assume a locale — weeks start Monday by default; set weekStart if the market needs Sunday

8Engineering

Extracted from the office-migration date & time workbench. The shared CSS owns presentation only — the calendar month maths, roving-tabindex keyboard model, forgiving date parsing, and popup/sheet open-close logic are behaviour the consuming platform implements. Acceptance criteria

  • One calendar surface backs all three modes; inline / input / sheet differ only in presentation
  • Correct month maths: real weekday offsets, Monday-first by default, today from the system clock
  • min/max are inclusive; disablePast keeps today selectable
  • WAI-ARIA grid with roving tabindex; arrows/Page/Home/End/Enter as documented; Escape closes overlays
  • Day cells, month nav, and the field toggle are all ≥44×44px
  • Typed entry parses day-first numerics, ISO, and month names; impossible dates raise the shipped Input error state
  • Selection is shared across modes; disabled days are aria-disabled and focusable, not disabled
  • prefers-reduced-motion: popup animation and transitions removed

9Open Questions

ℹ️
Ranges, multi-month, and ds-calendar v1.0 picks a single date. Date ranges (start→end), multi-month panels, and full month/agenda calendar views are out of scope — the latter is the deliberately deferred ds-calendar (its own design exploration). The weekStart prop is specced but Monday-only until a market needs Sunday. These extend the current class contract rather than break it.
