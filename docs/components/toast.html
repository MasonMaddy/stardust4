Component

Toast

WIP v1.0.0 Updated 2026-07-07

A transient, non-blocking notification that confirms an action or surfaces a status without interrupting the task. Toasts stack in a persistent polite live region, auto-dismiss after about five seconds with a visible progress bar that pauses on hover or keyboard focus, and cap the queue at three. It carries all four feedback severities — success, error, warning and info — where the legacy toast had only success and error.

4Usage Guidelines

✓ Do

  • Confirm a completed action (“Booking saved.”) or surface a passing status briefly
  • Keep the message to a short plain-text phrase that reads at a glance
  • Pick the severity that matches the outcome — success, error, warning or info
  • Let it auto-dismiss; rely on the pause-on-hover/focus so it can still be read
  • Dock to --top-right on desktop and --bottom on phones

✗ Don't

  • Use it for content that must be acknowledged — use a Dialog or Message Box instead
  • Put links, buttons or other interactive content in the message — a toast auto-dismisses out of reach
  • Rely on it to carry essential information a user could miss if they look away
  • Stack many at once — the queue caps at three for a reason; batch or summarise instead
  • Use an error toast for a blocking failure that needs a decision — that belongs inline or in a dialog

6Accessibility

Live regionThe stack is a permanent role="status" / aria-live="polite" region, present even when empty, so pushes are announced without stealing focus
Dismiss targetThe dismiss button’s visible glyph is 12px but its hit area expands to 44×44 via ::after — meets the touch-target minimum

Acceptance criteria

  • Four severities (--success/--error/--warning/--info) map to the feedback tokens; tint, border and progress bar match each severity
  • Stack is a permanent role="status" aria-live="polite" region, present when empty
  • Auto-dismisses after ~5s with a visible progress bar; hover or keyboard focus pauses it, and it resumes with the time left
  • Queue caps at three visible toasts — the oldest drops when a fourth is pushed
  • Dismiss button: aria-label, 44×44 hit area, focus-visible ring on colour/border/focus
  • Icon and progress bar are aria-hidden; meaning is carried by the message text, not colour alone
  • --top-right slides in from the right; --bottom docks bottom-centre above the safe-area inset and slides up
  • prefers-reduced-motion: enter/leave animation removed, progress bar frozen full, timer still runs

9Open Questions

ℹ️
Action affordance & assertive errors v1.0 is message-only and always polite. An optional single-action slot (e.g. “Undo”) and an assertive live-region mode for error severities are deferred — an action inside an auto-dismissing toast raises reachability questions, and assertive announcements can be disruptive. Both can be added without breaking the existing class contract if a flow needs them.
