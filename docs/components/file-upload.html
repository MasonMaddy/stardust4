Component

File Upload

WIP v1.0.0 Updated 2026-07-07

One consolidated uploader: a drag-and-drop zone that also accepts clicks and keyboard, with a file list that shows names, sizes, per-file progress, errors and removal. It consolidates the legacy InputFile (compact row) and DragAndDropFileUpload (dropzone) into a single component in two layouts — the soft-deprecated FileUpload is not ported. Its key improvement over the legacy trio: an invalid file-type drag gets a visible rejection with the reason, where the legacy uploaders silently ignored it.

    4Usage Guidelines

    ✓ Do

    • Use the dropzone layout for multi-file uploads (bulk enrolments, document batches)
    • Use the --compact layout when a single file sits inline in a form, like a field
    • Set accept so the native dialog and drag validation both filter to the right types
    • State the accepted types and any size limit in the requirements hint (--sub)
    • Keep the invalid-drag rejection visible — the reason is what the legacy uploaders never showed

    ✗ Don't

    • Silently ignore an unaccepted file — always name why it wasn’t added
    • Attach behaviour to a non-focusable zone — the real input must stay the keyboard target
    • Truncate the tail of a filename — middle-truncate so the extension survives
    • Use it for a single always-required avatar/logo crop — that’s an image-picker pattern, not this
    • Rely on colour alone for the error/invalid states — pair the tint with the reason text

    6Accessibility

    RequirementImplementation
    Keyboard targetA real input[type=file] is visually hidden but focusable and labelled — Tab reaches it, Enter/Space open the native dialog. The visible ring is :focus-within on the zone
    Touch targetThe remove button is a full 44×44 target (negative margins keep the row tight); on touch the whole zone is a large tap target for the native picker
    Live regionA permanent visually-hidden aria-live="polite" region announces adds, removals, completions and rejections without stealing focus
    Invalid feedbackAn unaccepted type is never silently dropped — the zone shows a visible reason and a dismissible role="alert" message names it
    ProgressThe bar is decorative (aria-hidden); start and completion are carried by the polite live region (“Added…”, “… uploaded.”), consistent with the sibling Toast. A platform build may instead expose a native progress semantics per the Engineering acceptance criteria
    Full filenameMiddle-truncation is visual only — the complete name stays in the DOM (and in title), so AT reads it in full
    IconsUpload / block / file glyphs are decorative — aria-hidden="true"; state meaning is carried by text, not colour or icon alone
    Contrast#252525 primary text on #FFFFFF ≈ 16:1; #D21D3E error text and the error tint meet WCAG AA for text and ≥3:1 for UI-component borders
    Focus indicatorZone ring (:focus-within) and remove-button ring (:focus-visible) both use colour/border/focus — never on :hover
    DisabledDisabled zone removes pointer, drop and keyboard interaction and mutes text to colour/text/disabled

    Acceptance criteria

    • One component, two layouts: dropzone and --compact input-like row; legacy FileUpload not ported
    • Real input[type=file] — visually hidden, focusable, labelled; Enter/Space open the native dialog; accept gates the picker
    • Drag: valid type → --dragover highlight; invalid type → --invalid with a visible reason and a dismissible message
    • File list: middle-truncating name (extension preserved), size/type meta, progress bar to 100% with “Uploaded”, per-file error, 44px remove
    • Adds, removals, completions and rejections announced via a polite aria-live region
    • Touch: drag affordance hidden ≤640px (or --touch); zone becomes a tap target
    • Zero hardcoded hex — all values reference --sd-* tokens; prefers-reduced-motion removes transitions and animated fill
