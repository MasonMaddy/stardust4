Component

Bottom Sheet

WIP v1.0.0 Updated 2026-06-12 DRI: Mason

A modal selection surface that slides up from the bottom edge — web only: native iOS and Android apps must use the platform sheet containers instead (Sections 4 and 5), never this component. For mobile web, PWAs and webviews, ds-sheet composes existing DS pieces: rows are slim selectable ds-cards (title-only Title Blocks), CTAs and the X close are ds-btn. Three dismiss patterns: instant auto-close, Confirm, and multi-select Apply (n).

MD3 scroll-aware dividers

ℹ️
How it works Header and footer each carry a permanent 1px transparent border, so nothing shifts. JS syncs two classes on .ds-sheet from the list's scroll position: .has-scroll-top (set while scrollTop > 1) colours the header's bottom border, and .has-scroll-bottom (set while content remains below the fold) colours the footer's top border with colour/border/default. The dividers therefore appear only while content is actually scrolled behind the header or footer — the MD3 pattern. Try it in the multi-select demo above: at the top of the list there is no header divider; scroll down and it fades in while the footer divider fades out at the end.

ℹ️
X-close rule The X close button is required whenever the sheet has a confirm CTA — it gives a visible, focusable dismiss that is clearly distinct from Confirm/Apply. On instant (auto-close) sheets it is optional, since picking a row already dismisses and scrim/Escape cover cancel.

4iOS — native usage

⚠️
Never ship ds-sheet in a native iOS app Use the platform sheet container — UISheetPresentationController (UIKit) or SwiftUI presentationDetents — and style the content inside it with DS tokens. The platform container gives you detent physics, drag-to-dismiss, keyboard avoidance, VoiceOver semantics and safe-area handling for free; a web reimplementation of any of these would always lag the OS.
  • Detents: .medium (half-height — the analogue of this component's quick-pick use) and .large. Selection sheets should open at .medium; offer .large only if the list is long.
  • Grabber: show it (prefersGrabberVisible = true / .presentationDragIndicator(.visible)) whenever the sheet is resizable or drag-dismissable — roughly 36×5pt, drawn by the system.
  • Selection rows: iOS has no dedicated multi-select sheet — compose a List with checkmarks (single) or toggled checkmarks (multi), usually with Cancel/Done in a header. Style the rows to the DS Card spec inside the system container.
  • iPad has no bottom sheets: the same presentation becomes a form sheet (centred card over a dimmed background) for focused self-contained tasks, or a popover (anchored to the trigger, no scrim) for transient pickers.
  • Date & time: always the native pickers (UIDatePicker / SwiftUI DatePicker) — per the Input spec. Never rebuild a date picker inside any sheet.

5Android — native usage

⚠️
Never ship ds-sheet in a native Android app Use Material 3's ModalBottomSheet (Jetpack Compose) and style the content inside it with DS tokens. The 28dp top radius, the 32×4dp drag handle, the scrim, drag physics, back-button dismissal and TalkBack semantics all come free with the platform container.
  • Container: ModalBottomSheet { … } — keep the system drag handle and scrim; put DS-styled selection rows (Card spec) inside.
  • Selection rows: M3 has no dedicated multi-select sheet either — compose checkbox/radio list items inside the sheet, mirroring this component's instant / confirm / Apply (n) patterns.
  • Width cap: M3 caps the modal sheet at 640dp and centres it automatically on large screens — the behaviour .ds-sheet--centered mirrors on web. No extra work needed.
  • Tablets: the capped, centred sheet remains valid; for filters and secondary tasks that should sit alongside content, prefer a modal side sheet (right-anchored, 256–400dp wide).
  • Date & time: the native M3 date/time pickers — per the Input spec.

Do / Don't

✓ Do
Reserve the sheet for mobile-web viewports (and PWAs/webviews) where the thumb-reach, bottom-anchored pattern earns its place — and pick the lightest dismiss pattern the choice allows: instant for low-risk picks, Confirm only when reviewing matters.
✗ Don't
Never use a sheet on desktop web — a bottom-anchored surface on a large pointer-driven screen is the wrong pattern. Use the Phase 2 Select dropdown (or a dialog) instead.
✓ Do
One sheet at a time. Closing one task before opening the next keeps focus management predictable — the sheet always returns focus to its trigger.
✗ Don't
Don't nest or stack sheets. A sheet opening another sheet buries the user two modal layers deep with two scrims and broken focus return. If a row needs a follow-up step, close the sheet and navigate, or rethink the flow.

Content

  • The header title names the task ("Dietary needs", "Pickup person") — it is the dialog's accessible name
  • Rows are title-only: no avatars, subtitles, or pills inside sheet rows — the slim 14px row exists so more options fit above the fold
  • Keep CTAs to verbs: Confirm, Apply (n); the count in Apply updates live with the checked rows

8Accessibility

💬
Dialog semantics
The sheet element carries role="dialog" aria-modal="true" and is named by its title via aria-labelledby. The grabber is decorative (aria-hidden="true") — drag is not a web affordance in v1.
🃏
Row semantics inherit Card
Rows are whole-card controls per the Card spec (review D2): role="radio" with roving tabindex inside role="radiogroup" (arrows move + select, Space selects), or independent role="checkbox" rows (Space toggles). Glyphs are aria-hidden; state is announced from aria-checked.
🎯
Focus management
On open, focus moves to the sheet (tabindex="-1" on .ds-sheet); on close — by any dismiss path — focus returns to the trigger that opened it.
⌨️
Keyboard dismiss
Escape closes the sheet from anywhere inside it. The X close (where present) is a real focusable button with aria-label="Close".
⚠️
Known v1 gap: no full focus trap The demos on this page (and the v1 reference behaviour) move focus to the sheet and back to the trigger, but do not trap Tab — a keyboard user can Tab out of the open dialog into the page behind the scrim, which contradicts aria-modal="true". This is recorded honestly as an engineering acceptance criterion (Section 9): production implementations must cycle Tab/Shift-Tab within the sheet.
ℹ️
Keyboard avoidance — deferred Sheets containing text inputs (e.g. a combobox row filter) need on-screen-keyboard avoidance. That work is deferred to Input Phase 3; v1 sheets contain selection rows and buttons only.

interface SheetProps {
  variant?:  SheetVariant; // default: 'instant'

  title:     string;       // header title — the dialog's accessible name

  onPick?:   (id: string) => void;

  onApply?:  (ids: string[]) => void;

Acceptance criteria (web)

  • Web only: the component is never rendered inside native iOS/Android apps (Sections 4–5)
  • Overlay/sheet semantics: role="dialog" aria-modal="true" aria-labelledby → title; grabber aria-hidden
  • Full focus trap: Tab/Shift-Tab cycle within the open sheet — closes the documented v1 demo gap
  • Focus moves to the sheet on open and back to the trigger on close, on every dismiss path
  • Dismiss matrix per Section 3: scrim click, Escape, X close (required with a CTA), instant variant auto-closes ~250ms after pick
  • Rows are selectable ds-cards per the Card spec: roving-tabindex radio groups, Space-toggled checkboxes, glyphs aria-hidden
  • MD3 scroll dividers: .has-scroll-top/.has-scroll-bottom synced from the list's scroll position; 1px border/default, no layout shift
  • Safe area: footer bottom padding includes env(safe-area-inset-bottom)
  • Heights: max-height: 75%; centered caps at min(640px, 100% − 32px) and centres
  • Motion: slide-up at motion/duration/slow + easing/standard; disabled under prefers-reduced-motion
  • Multi: Apply (n) live count tracks checked rows; Clear unchecks all and resets the count
  • All token values match Section 7 — no hardcoded hex (including the new colour/scrim)
