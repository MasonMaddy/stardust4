Component

Input

WIP — Phase 3 v1.3.0 Updated 2026-06-12 DRI: Mason

A unified input shell covering text, password, search, send, and textarea variants (Phase 1), plus select, multi-select with chips, and combobox (Phase 2). Phase 3 adapts those to mobile: under 768px they open a Bottom Sheet instead of the dropdown, and date/time fields wrap the native OS pickers. File variants follow in later phases. All variants share identical visual states, tokens, and label behaviour.

4Mobile (Phase 3)

ℹ️
Breakpoint rule < 768px → the dropdown becomes a Bottom Sheet; ≥ 768px → the Phase 2 dropdown. Detect by viewport width / coarse pointer media query — never user-agent sniffing. The field itself is unchanged on both sides of the breakpoint; only the overlay swaps.

Under the breakpoint, tapping a select/multi-select/combobox field opens a ds-sheet instead of the anchored ds-menu. Picking writes the value back to the field display; multi-select writes chips back on Apply (chips are the Pill component, ds-pill --sm --green --minimal). The combobox pins a search Input in the sheet (.ds-sheet__search). Date and time fields never open a sheet — they wrap the native OS pickers: the DS styles the field, the OS owns the picker UI.

Keyboard avoidance

Sheets that contain inputs (the combobox search) must shrink above the on-screen keyboard rather than being covered by it. Implementation: listen to the visualViewport resize event and constrain the sheet to the visible viewport height (or use the interactive-widget=resizes-content viewport meta setting where supported). The target behaviour can be previewed in the sandbox, which mocks a native keyboard sliding up under the educator sheet — the mock itself is a sandbox-only artifact and is not part of the component.

Engineering notes — mobile

  • useIsMobile() composable — single source of the breakpoint decision: matchMedia('(max-width: 767px)') width check and/or (pointer: coarse). Never user-agent sniffing.
  • Same props, different overlay — DsInputSelect / DsInputCombobox / multi-select swap the ds-menu dropdown for a ds-sheet under the breakpoint; the public props/events interface is unchanged (see Engineering).
  • 16px field font — the .ds-input__field 16px font-size rule (Anatomy part 5, acceptance criteria) is what prevents iOS auto-zoom when a sheet's search input is focused — do not shrink it on mobile.
  • Sheet behaviour — dismiss patterns, scroll-aware dividers, focus handling and the full behaviour table live in the Bottom Sheet spec; this section only covers the Input ↔ sheet glue.

5Usage Guidelines

Label placement — outside vs inside

✓ Do — outside label
Label always visible. Passes WCAG SC 1.3.5 (Input Purpose). Works with autofill and screen readers without special handling.
✗ Don't — inside label (new work)
Label disappears once user types, making the field's purpose ambiguous. Fails SC 1.3.5 for long-form inputs.
Service name

Use this component when

  • Collecting user text input in a form
  • Providing a search interface (type=search)
  • Building a chat-style send interaction (type=send)
  • Multi-line free text (type=textarea)

Don't use this component when

  • Choosing from a fixed list — use the Select / Combobox variants (DsInputSelect / DsInputCombobox, Phase 2 — see Variants above)
  • Picking a date or time — on mobile, wrap the native OS pickers (<input type="date|time"> as the field — see Mobile (Phase 3)); the desktop web date pattern lands in a later phase
  • Toggle a boolean setting — use Toggle

Helper text

  • Use for format hints, constraints, or supplemental guidance ("dd/mm/yyyy", "Min 8 characters")
  • Error message replaces helper text — don't show both simultaneously
  • Keep it to one line; wrap if necessary but avoid multi-paragraph

Placeholder vs label

  • Placeholder is supplemental — it disappears when the user types. Never use it as the sole label.
  • If you have both a label and a placeholder, the placeholder should show an example value ("E.g. Ada Lovelace"), not repeat the label

7Accessibility

🏷️
Label association
Always pair <label for="…"> with <input id="…"> or wrap the input in the <label>. Never use placeholder as the sole label — it is not announced as the accessible name by all screen readers, and disappears when the user types.
📐
Touch target
Default height 56px exceeds iOS 44pt and Android 48dp minimums. Comfortable (48px) meets the minimum. Compact (40px) is below the Android recommendation — use sparingly. Clear button is 24×24 visual but wraps in a 44×44 tappable area in native implementations.
🎨
Colour contrast
Input text (text/primary #252525 on white): 16.1:1 — AAA.
Placeholder (text/secondary #838383 on white): 4.48:1 — meets AA for large text; borderline for 16px normal weight. Placeholder is supplemental only — never the sole label.
Error border (#D21D3E on white): passes non-text contrast 3:1 (WCAG 1.4.11).
Focus ring (#DFF2F1 on white): supplemental — the border colour change carries the primary error/focus signal for users who can't perceive colour alone.
⌨️
Keyboard
Tab moves focus into the field. Clear button and action button are in the natural tab order. Search: Enter in the field fires the @search event. Send: Enter fires @send when the field is non-empty. Password eye toggle: Space / Enter toggles visibility; aria-label updates to "Show password" / "Hide password".
❗
Error announcements
Connect the helper/error text to the input via aria-describedby. When the error state activates, set aria-invalid="true" on the input. The helper text element should have a stable ID so screen readers can follow the association.
🔊
Screen reader
Web Native <input> / <textarea> — all roles and states built-in.
iOS Use UITextField / UITextView with accessibilityLabel set from the outside label prop.
Android Use TextInputLayout / EditText with contentDescription and hint set appropriately.
📋
Select / Combobox — ARIA pattern (Phase 2)
Follows the WAI-ARIA combobox/listbox pattern: the trigger (select box or combobox text field) carries role="combobox" with aria-haspopup="listbox", aria-expanded (synced open/closed), and aria-controls pointing at the menu. The menu is role="listbox" (aria-multiselectable="true" for multi select), labelled via aria-labelledby; each option is role="option" with aria-selected. Chip remove buttons carry per-value aria-labels ("Remove …").
⌨️
Select / Combobox — keyboard map (Phase 2)
Enter / Space / ArrowDown open the closed menu. ArrowDown / ArrowUp move the active option. Enter picks the active option. Escape closes the menu; clicking outside also closes it. In the combobox, typing filters the option list.
⚠️
Inside label fails SC 1.3.5 The inside label (floating-label pattern) violates WCAG 2.1 Success Criterion 1.3.5 (Identify Input Purpose) in many implementations because the label is replaced by the value once filled, making it impossible for assistive technology to identify the purpose from a visible label. This is one of the primary reasons it is deprecated.

9Engineering

ℹ️
Architecture note DsInput is the base shell. Each named variant (DsInputSearch, DsInputSend, DsInputTextarea etc.) is a thin wrapper that passes a type prop and wires up the appropriate event/button logic. Phase 2 adds DsInputSelect and DsInputCombobox on top of the same shell. Phase 3: under the 768px breakpoint (useIsMobile() — width / coarse-pointer media query, no UA sniffing), select, multi-select and combobox render the Bottom Sheet variant instead of the dropdown — same props and events, different overlay. No new public API; the overlay swap is internal to the components.

Acceptance criteria

  • All 5 states render correctly: unfilled, focus, filled, error, disabled
  • 3 densities: 56px, 48px, 40px
  • Focus ring appears on :focus-within only — never on hover
  • Outside label: always visible, properly associated via for/id
  • Inside label: renders but emits console.warn in dev builds
  • Error state: sets aria-invalid="true" and aria-describedby on helper text
  • Clear button: hidden when empty; visible when filled; fires @clear and clears model value
  • Password: eye button toggles type; aria-label updates to match current state
  • Search: Enter and search button both fire @search with current value
  • Send: send button disabled when empty; Enter and button fire @send
  • Textarea: height auto, min-height --input-height, resizable vertically
  • 16px font-size on <input> prevents iOS auto-zoom
  • All token values match Section 6 tables

10Open Questions

⚠️
1 · Inside label sunset timeline The insideLabel prop is marked deprecated in this spec. A 6-month removal window is recommended (i.e., remove in the next major version after all PHP/Vue3 usages have been migrated to outside label). Confirm the timeline with the PHP team before setting a hard removal date.
ℹ️
2 · Figma gap — mobile bottom sheet overlays not yet designed Partially resolved 2026-06-12: the sheet patterns are now defined by the built Bottom Sheet component and documented in Mobile (Phase 3) — implementation is no longer blocked. Figma frames for the mobile overlays are still wanted for design parity (the Bottom Sheet's own Figma comparison is also pending).
ℹ️
3 · Compact density below Android touch minimum Compact (40px) is 4dp below the Android 48dp minimum touch target recommendation. Consider whether compact density should be restricted to non-interactive contexts (e.g. read-only / display-only fields) or supplemented with invisible padding.
ℹ️
4 · Filled state border — no Figma differentiation from Unfilled The Figma component shows the same border/strong (#838383) for both unfilled and filled states. The only visual change when filled is input text colour switching from placeholder grey to text/primary. This matches the Figma spec but differs from some other design systems. Confirm this is intentional.
ℹ️
5 · Required field indicator No asterisk or "Required" label is defined in the Figma component. Confirm whether required fields should show an asterisk (*) appended to the label, a "(required)" text, or remain unmarked in favour of marking optional fields instead.
ℹ️
6 · Select / Combobox on mobile — bottom sheet adaptation is Phase 3 Resolved 2026-06-12: Phase 3 has landed — see Mobile (Phase 3). Under 768px, Select / multi-select / Combobox open the Bottom Sheet; the Phase 2 dropdown remains the ≥ 768px pattern. Figma frames for design parity are tracked in question 2.
ℹ️
7 · Desktop web date pattern Date and time on mobile are always the native OS pickers (decided 2026-06-12 — never a sheet). The desktop web date pattern is not yet in the library and will be added later by Mason.
