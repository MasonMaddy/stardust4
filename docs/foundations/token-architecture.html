Token Architecture

How Stardust's design tokens are structured: a two-tier model where components reference purpose-named aliases, never raw values.

The two-tier model

TierWhat it isUsed in components?
PrimitiveRaw values, named by what they are (e.g. a colour scale step, a spacing step). Never referenced directly.No
SemanticIntent-based aliases that reference primitives (e.g. action/primary, surface/default, text/primary).Yes — always

A component references the semantic layer, so when a primitive value changes on a Figma re-sync, every component that uses the alias updates at once — with no edits.

Naming is the contract

In code, each Figma token path maps to a CSS custom property: replace / with - and prefix with --sd-.

colour/action/primary    →  --sd-colour-action-primary
spacing/component/md     →  --sd-spacing-component-md
radius/m                 →  --sd-radius-m

Names never change once in use — they are the stable contract between Figma, the CSS, and every consumer. Values may change on sync.

Live values

ℹ️
Stardust uses Legacy Stardust values today. The live primary action colour is teal #00776B (--sd-colour-action-primary). Coral #FF5A35 is the Xplor corporate brand used for the documentation site chrome only (--xp-*) — it is not a Stardust component token.

The authoritative values live in docs/assets/css/tokens.css and are documented on the foundation pages: Colour, Typography, Spacing, Radius, and Motion.

Machine-readable export (DTCG)

The full token set is published in the W3C DTCG format at a stable URL, derived from tokens.css and CI-guarded against drift:

https://masonmaddy.github.io/stardust4/tokens/stardust.tokens.json

Consume it with any DTCG-aware tool (e.g. Style Dictionary v4 to generate Swift, Compose/Kotlin, or XML). It is a committed source file (stardust.tokens.json) served verbatim — nothing is generated at request time. See the Getting Started guides for per-platform consumption.
