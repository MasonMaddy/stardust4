Foundations

Philosophy

The principles behind Stardust — how the system is structured, why it's built the way it is, and what stays true as it grows.

Principles

  • One source of truth per concern. Tokens live in one place, each component's styles live in one place, and the docs render from those — never a second copy that can drift.
  • Tokens are the contract. Token names are stable; values may change on a Figma re-sync, names never do. Components reference token names, so a value change flows everywhere at once.
  • Component CSS is the single source of truth. Doc pages link the shared stylesheet from assets/css/components/ — they never re-declare a component's rules.
  • No build step. The site is plain HTML/CSS/JS served as-is. Simplicity and longevity over tooling.
  • Designed for people and agents. The same system is consumed by designers (Figma), engineers (per-platform code), and AI agents (the machine-readable Component API).
  • Accessibility is non-negotiable. Touch targets, focus visibility, and semantics are part of every component, not an afterthought.

The chain of truth

Each layer derives from the one before it, so there's a single place to change any given thing:

LayerSource of truth
Design tokensdocs/assets/css/tokens.css (synced from Figma) → Token Architecture
Component stylingdocs/assets/css/components/<name>.css — linked by doc pages
Human docsThe component and Foundations pages you're reading
Machine-readable exportsDTCG tokens + Component API — both derived, CI-guarded

How we work

New components move through a defined pipeline — review against references, spec, iterate in a sandbox, pass an independent quality gate, then build the shared CSS and doc page. See the Prototype Workflow and Handoff Standard in the Playbook.
